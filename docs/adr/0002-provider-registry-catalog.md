# 0002. Provider Registry Catalog

## Context

Hoist must let users add keys for the providers their agent harnesses (Claude
Code, OpenCode, Codex) and routers (LiteLLM) accept. The union across those
catalogs is large: paseo 6 + ~35, OpenCode/models.dev 144, LiteLLM ~135. ~90% of
the long tail is uniform (`<PROVIDER>_API_KEY`, OpenAI-compatible base URL,
Bearer, probe `GET {base}/models`). A minority need hand-coding: cloud-credential
auth (Bedrock/Vertex/Azure/Cloudflare), OAuth (Copilot/GitLab/OpenAI/xAI), local
no-auth (Ollama/LMStudio), non-standard tokens (`HF_TOKEN`).

## Decision

Vendored catalog at `src/shared/providers/catalog.ts`, generated by
`scripts/gen-catalog.ts` from checked-in source snapshots. The generator merges
paseo built-ins + models.dev + LiteLLM majors, then **deduplicates and
canonicalizes** (e.g. `zai`/`zhipuai`/GLM collapse to one entry with `aliases[]`;
`alibaba`/`dashscope`/`qwen` likewise).

```ts
interface ProviderEntry {
  id: string
  label: string
  aliases?: string[]
  featured?: boolean
  envKeys: string[]
  baseUrlEnv?: string
  authType: 'api_key' | 'oauth' | 'cloud_creds' | 'none'
  probeKind: ProbeKind
  budgetProbeKind?: BudgetProbeKind
  defaultBaseUrl?: string
  notes?: string
}
```

~35 entries are `featured: true` for the picker's default view; the full ~150 is
searchable. A **custom OpenAI-compatible** provider (base URL + key + optional
headers) covers anything missing. One provider can have multiple **profiles**
(`claude-work`, `claude-personal`), mirroring paseo's `extends` model. Config
validated with Zod.

## Consequences

- Positive: one source of truth, regenerable from upstream; custom escape hatch.
- Negative / trade-offs: generator must track upstream drift; canonicalization
  rules need maintenance; ~150 entries inflate the picker unless filtered.
- Follow-ups: version-pin upstream snapshots; canonicalization tests;
  "missing provider?" affordance.
